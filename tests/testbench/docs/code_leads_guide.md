# 代码线索 (开发者) 使用说明

本页面向**记忆系统的代码相关人员**, 讲清「代码线索」子页是什么、一条线索该怎么读、它检查了哪些东西、以及**在哪里它就不再可靠**。普通测试员可以忽略本页。

> **一句话**: 这些是「值得去主程序代码里排查的**方向 / 线索**」, **不是**已经确认的 bug 报告。由记忆数据反推代码问题**并不完全可靠**, 每一条都需要你人工确认后才能下结论。

---

## 1. 它是什么、为什么可信度有限

testbench 的记忆分析看到的是**磁盘上的数据形状**: 某角色此刻的 facts / reflections / persona / 对话在盘上长什么样。它**看不到运行期过程**: 抽取时喂了什么、embedding 何时跑、写入调用栈是谁。

「反推代码缺陷」本质是从**数据的异常**倒推**代码的 bug**。这条链条只有在「异常与某段代码路径存在近乎唯一的因果对应」时才靠得住:

- 对**结构完整性**类异常 (悬空引用、向量损坏、主键重复……), 这种对应通常成立 —— 违反几乎一定是写入 / 迁移代码的问题。所以本页**只反推这一类**。
- 对**内容质量**类问题 (矛盾、冗余、归因漂移……), 同一个「坏现象」可能来自模型能力、用户数据、配置阈值等**多个非代码因素**, 反推会退化成猜测。所以本页**刻意排除这一类** (见 §4)。

即便是被反推的这一类, 目前也只能给到**导航级弱线索** —— 「这类不变量被违反了, 建议排查主程序某模块的写入 / 迁移路径」, 而**不能**自动定位到具体某一行代码。

---

## 2. 在哪里、怎么用

1. 打开顶部 **「记忆系统分析」** workspace。
2. 选好角色, 切到第 4 个子页 **「代码线索 (开发者)」**。
3. 页面顶部是一块**不可关闭的红色警告** —— 务必先读, 它重申了本页的定位与边界。
4. 下方按需点 **[重新分析]** 重新拉取。

本页是**纯读**操作: 不取会话锁、不触发自动保存、不调用任何大模型, 也**绝不会**修改任何代码 —— testbench 全程不写主程序。

---

## 3. 一条线索怎么读

每条线索卡片包含:

- **线索代码 + 强度标记**: 强度分「较强 / 中等 / 较弱」。
  - **较强** = 违反几乎必是写入 / 迁移代码问题;
  - **较弱** = 也可能只是某道工序还没跑, 未必是代码信号。
  - 无论强弱, 都带一条固定的 **「(仍需人工确认)」**。
- **违反了什么**: 用一句话说明命中的不变量, 以及命中处数。
- **建议排查 (祈使句)**: 一个排查动作, 而**不是**结论。卡片刻意不使用「确认存在 bug / 缺陷」这类断言。
- **疑似方向**: 主程序里值得先看的模块 —— 是**方向**, 不是精确定位。
- **还需哪些运行期证据才能坐实**: 明确告诉你, 仅凭当前磁盘数据还不足以定案, 缺的是什么 (写入调用栈 / embedding 轨迹 / 跨快照 diff 等)。

## 4. 都检查了哪些 (机械不变量)

本页只把下列**结构完整性 / 机械不变量**类发现反推成线索:

| 线索 | 现象 | 建议排查方向 |
|---|---|---|
| 悬空来源引用 | 反思声明的来源事实已被硬删除 | 删除事实时是否级联清理了引用它的反思 |
| 断裂晋升 | 有晋升人设标记来自反思, 却没有对应的晋升 / 合并溯源边 | 反思晋升 / 合并写入时来源边的赋值是否遗漏 |
| 游离反思 | 有反思没有任何有效来源事实 | 反思合成时来源事实的绑定逻辑 (较弱: 也可能本就无源) |
| 向量过期 | 向量与其当前文本的 sha256 不一致 | 记忆文本更新后是否触发了向量重算 |
| 向量损坏 | 向量维度或数值非法 | 向量写入前的维度 / 数值校验点 |
| 多向量空间 | 混入了不同维度 / 模型的向量 | 是否在同一角色里混用了不同的嵌入模型 / 维度 |
| 重复主键 (ID-DUP) | 同一记忆文件里出现重复主键 `id` | 该记忆容器的主键分配与写入去重逻辑 |
| 重复事件 (EVT-DUP) | `events.ndjson` 里出现重复 `event_id` | 事件 append / reconcile 路径是否重复写入了同一 `event_id` |

> 这些「疑似方向」是排查起点, 不是精确定位。真正定案仍需你到主程序对应模块里核对。

---

## 5. 为什么「内容质量类」不在此反推

矛盾、冗余重复、归因质量、晋升语义漂移、流水线比率、留存质量等发现, **刻意不做代码反推**。它们是**数据 / 模型 / 配置**层的产物:

- 「重复多」多半是相似度阈值设置, 或用户反复说同一件事;
- 「矛盾」多半是用户改主意或模型抽取不一致;
- 「晋升语义漂移」可能本就是设计允许的抽象改写。

把它们当代码缺陷会制造大量假阳性, 误导你去改**没有 bug** 的代码。这类问题请走**记忆运营 / 配置调参** (在「系统概况」里查看), 而不是改代码。本页底部的**「以下类别故意不做代码反推」**区会如实列出这次跳过了哪些、各多少项。

---

## 6. 检查状态 (诚实回报, 从不装作没问题)

页面底部的**检查状态**区会明确区分「**已运行**」与「**未检查**」, 绝不把「没检查」伪装成「没问题」:

- **向量检查**: 该角色若没有可用向量, 会显示「未检查」—— 这**不代表**向量一定没问题, 只是这次没法查 E2 / E3 / E4。
- **事件流检查**: 若没有 `events.ndjson` (导入的真实角色常常不带), 会显示「未检查」; 事件行数过多触发上限时显示「已截断」。

> 记住: **没有线索 ≠ 代码没 bug**。可能只是相关检查这次没跑起来 —— 状态区就是用来告诉你这一点的。

---

## 7. 局限 (务必知晓)

- **这是导航级弱线索, 不是定位器**: 最多告诉你「哪类不变量被违反、值得看哪个模块」, 不能自动指到具体某一行。
- **可能假阳性**: 数据 / 模型 / 配置原因都可能触发, 未必是代码。
- **只读边界**: 本页不修改任何代码, 也不写主程序 —— 所有修复都由你手动完成。
- **样例仅供本地查看**: 卡片里的少量 id / 标签样例仅方便你在本地定位, 请勿截图外传。

## 8. 常见问题

- **Q: 这一页会不会改动主程序或我的会话?**
  不会。纯读、不写盘、不取锁、不 autosave、不调模型, 更不碰主程序代码。

- **Q: 一条线索也没有, 是不是就说明代码没问题?**
  不一定。先看 §6 的检查状态: 如果向量 / 事件流是「未检查」, 说明相关检查这次没跑, 不能据此下「无 bug」结论。

- **Q: 我看到一条「较强」线索, 能直接提 bug 吗?**
  请先到卡片指出的**疑似模块**里人工确认, 并补齐「还需哪些运行期证据」里列出的信息, 确认后再提。

- **Q: 内容质量类的问题我也想追代码, 怎么办?**
  这类反推不可靠 (见 §5)。建议先从数据 / 阈值 / 模型角度排查; 若确要追代码, 需要主程序侧补运行期证据 (写入调用栈 / 结构化事件钩子) 后再做, 而不是仅凭当前磁盘数据。
